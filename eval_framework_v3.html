0. 评估场景定义

被评估的 Agent 接收一封或多封邮件，需要完成以下端到端流程：

  1. Context 获取：从邮件中提取关键信息，必要时调用工具（日历、通讯录、知识库等）补充上下文。
  2. ToDo List 生成：将邮件意图拆解为一个有序的任务列表。
  3. Deliverable 产出：对每个 ToDo 项执行任务并生成对应的交付物（回复邮件草稿、会议邀请、文档、报告等）。

评估分为三个模块：

模块评估对象核心问题权重
A. ToDo List 识别任务拆解结果识别了哪些任务？顺序和属性对不对？0.30
B. ToDo 执行质量每个 ToDo 的端到端执行做成了没有？过程对不对？产出好不好？0.50
C. 稳定性重复运行与扰动测试结果一致吗？扛不扛得住干扰？0.20

1.3 标注流程

每个 test case 的标注步骤（预计 10-15 分钟/case）:

Step 1: 人工标注 gold_todos（含依赖关系和属性）
        → 覆盖 M1 Task F1, M2 Edge F1, M3 Attribute Accuracy
        
Step 2: 人工标注 gold_tool_calls
        → 覆盖 M6 Tool Invocation Quality

Step 3: LLM 自动生成 gold_key_info 初稿 → 人工审核修正
        → 覆盖 M5 Information Utilization

Step 4: LLM 自动生成 rubrics 初稿 → 人工审核修正
        → 覆盖 M8 Content Quality

一次性定义（做一次全局复用）:

Step 0: 定义 task_type_rules（各任务类型的成功条件 + 格式检查）
        → 覆盖 M4 Execution Success, M7 Format Compliance
LLM 辅助标注 Prompt 示例（Step 3 — 生成 gold_key_info）：

给定以下邮件和 ToDo 任务，请列出执行该任务时可以利用的所有关键信息（来自邮件正文、附件、项目描述、工具返回结果等），每条信息用"字段名=值"的格式。

LLM 辅助标注 Prompt 示例（Step 4 — 生成 rubric）：

给定以下邮件和 ToDo 任务，请生成一份交付物评分 rubric。列出 4-6 个评分项，每项 1-3 分，总分 10 分。评分项应覆盖：信息利用充分度、任务完成度、准确性、专业性。

2. 评估数据集有效性验证

在正式评估 Agent 之前，必须先验证评估数据集本身的质量。

2.1 代表性（Representativeness）参考 GDPval [4] O*NET 覆盖率方法

验证数据集是否覆盖了真实邮件场景的分布。具体检查：场景覆盖率（日常沟通、会议安排、项目协作、审批流程、紧急事项等）、邮件复杂度分布（单任务/多任务/嵌套依赖）、语言风格多样性（正式/非正式、长/短邮件）。

$$Coverage = \frac{|\text{实际覆盖的场景类别}|}{|\text{预定义的全部场景类别}|}$$

目标：Coverage ≥ 0.85。

2.2 一致性（Inter-Annotator Agreement）Fleiss' Kappa — 标准统计方法

由至少 3 名标注员独立为同一批邮件标注 ground truth。计算 Fleiss' Kappa：

$$\kappa = \frac{\bar{P} - \bar{P_e}}{1 - \bar{P_e}}$$

其中 $\bar{P}$ 是观测一致率，$\bar{P_e}$ 是随机一致率（各类别总体选择比例的平方和）。标准：κ ≥ 0.7 为可接受，≥ 0.8 为良好。

2.3 可区分性（Discriminability）经典测验理论 Item Discrimination Index

使用历史迭代版本（不同 prompt 版本、不同架构版本）充当"不同考生"。将各版本按总分排序，取上下各 27%：

$$D_i = P_{upper} - P_{lower}$$

$D_i \geq 0.3$ 表示该用例具有良好区分度。

2.4 数据集有效性综合分数

$$DatasetValidity = 0.35 \times Coverage + 0.35 \times \kappa_{norm} + 0.30 \times \bar{D}$$

其中 $\kappa_{norm} = \frac{\kappa + 1}{2}$。DatasetValidity ≥ 0.75 方可进入正式评估。

3. 评估模块 A — ToDo List 识别

3.1 评估数据

  • 输入：email + context
  • Agent 输出：结构化 ToDo list
  • Ground Truth：gold_todos

3.2 评估指标

M1: 任务识别 F1（Task Identification F1）参考 TaskBench [1] Node F1

评估 Agent 识别任务的准确性和完整性。匹配判定使用 embedding cosine similarity ≥ 0.85。

$$Precision = \frac{|T_{pred} \cap T_{gold}|}{|T_{pred}|} \qquad Recall = \frac{|T_{pred} \cap T_{gold}|}{|T_{gold}|}$$ $$F1 = \frac{2 \times Precision \times Recall}{Precision + Recall}$$

报告中同时展示 Precision（多余任务惩罚）和 Recall（遗漏任务惩罚）的 breakdown。

所需数据：gold_todos[].description vs Agent 输出的 ToDo descriptions。

M2: 依赖关系 Edge F1参考 TaskBench [1] Edge F1

将任务间的依赖关系建模为有向图，比较预测图和 gold 图的边集。

$$EdgeF1 = \frac{2 \times |E_{pred} \cap E_{gold}|}{|E_{pred}| + |E_{gold}|}$$

所需数据：gold_todos[].depends_on vs Agent 输出的依赖关系。

M3: 属性准确率（Attribute Accuracy）参考 TaskBench [1] Parameter F1 思路

对匹配成功的任务，逐属性检查：

$$AttrAcc = \frac{1}{|M|} \sum_{i \in M} \frac{|\text{正确属性数}_i|}{|\text{总属性数}_i|}$$

所需数据：gold_todos[].priority, .deadline, .depends_on。

3.3 模块 A 综合分数

$$S_A = 0.45 \times F1 + 0.30 \times EdgeF1 + 0.25 \times AttrAcc$$

4. 评估模块 B — ToDo 执行质量（端到端）

对每个 ToDo item，从"信息获取 → 工具执行 → 交付物产出"作为一条完整链路评估。三个层级由外到内：

层级核心问题评估方法
L1: 执行结果这个任务做成了没有？Code-based 二元判定
L2: 过程质量执行过程对不对？Code-based 指标
L3: 交付物质量产出的东西好不好？Per-case rubric + LLM-as-Judge

4.1 L1: 执行结果

M4: 任务执行成功率（Execution Success）参考 Survey [5] Task Completion / Success Rate; GDPval [4] task success

每个 ToDo 的执行结果为二元判定，判定条件由 task_type_rules 定义：

$$ExecSuccess = \frac{|\text{执行成功的 ToDo 数}|}{|\text{总 ToDo 数}|}$$

所需数据：task_type_rules（全局复用，无需 per-case 标注）。

4.2 L2: 过程质量

M5: 信息获取充分度（Information Utilization）参考 Survey [5] Memory & Context Retention; ATOD-Eval [3] long-horizon memory

检查 Agent 是否充分利用了可用信息源：

$$InfoUtil = \frac{|\text{实际利用的关键信息}|}{|\text{该 ToDo 可用的关键信息总数}|}$$

所需数据：gold_key_info（LLM 生成 + 人工审核）。

M6: 工具调用正确率（Tool Invocation Quality）参考 T-Eval [7] step-by-step tool eval; TaskBench [1] Parameter F1 + NED

综合评估工具选择、参数传递和调用顺序：

$$ToolQuality = \frac{1}{|C|} \sum_{c \in C} \left( 0.4 \times sel_c + 0.4 \times param_c + 0.2 \times seq_c \right)$$
子项含义计算方式来源
$sel_c$工具选择正确选对工具为 1，否则为 0Survey [5] Tool Selection Accuracy
$param_c$参数正确参数名称和值的 F1TaskBench [1] Parameter Name F1
$seq_c$调用顺序正确$1 - \frac{EditDist(pred, gold)}{max(|pred|, |gold|)}$TaskBench [1] NED

所需数据：gold_tool_calls（人工标注）。

4.3 L3: 交付物质量

M7: 格式正确性（Format Compliance）参考 Survey [5] Code-based Metrics Computation

$$FormatScore = \frac{|\text{通过的格式检查项}|}{|\text{总格式检查项}|}$$

所需数据：task_type_rules[].format_checks（全局复用）。

M8: 内容质量 — Per-Case Rubric参考 GDPval [4] expert rubric grading; Survey [5] LLM-as-a-Judge

每个 test case 附带专属的评分 rubric，LLM-as-Judge 对照 rubric 逐条评分：

$$ContentQuality = \frac{\sum_{r} score_r}{total\_points}$$

所需数据：rubrics（LLM 生成 + 人工审核）。无 rubric 时使用通用 5 维度评分兜底（事实准确性 0.30 / 完整性 0.25 / 相关性 0.20 / 专业性 0.15 / 可操作性 0.10）。

4.4 Per-ToDo 综合分数

$$S_{todo_j} = 0.30 \times ExecSuccess_j + 0.30 \times ProcessQuality_j + 0.40 \times DeliverableQuality_j$$
$$ProcessQuality_j = \begin{cases} 0.5 \times InfoUtil_j + 0.5 \times ToolQuality_j & \text{需要工具调用} \\ InfoUtil_j & \text{不需要工具调用} \end{cases}$$
$$DeliverableQuality_j = 0.25 \times FormatScore_j + 0.75 \times ContentQuality_j$$

对于 task_type=tool_action 且无独立 deliverable 的 ToDo（如预订会议室），$DeliverableQuality_j$ 取 $ExecSuccess_j$ 的值。

4.5 模块 B 综合分数

$$S_B = \frac{1}{|T|} \sum_{j \in T} S_{todo_j}$$
完整示例 — Alice 邮件的四个 ToDo
ToDo T1: 查日历确认周五有空  (tool_action)
├── L1 执行结果: 1.0 (check_calendar 返回成功)
├── L2 过程质量: 0.5 × 1.0 + 0.5 × 1.0 = 1.0
└── L3 交付物质量: = ExecSuccess = 1.0 (无独立 deliverable)
→ S_T1 = 0.30×1.0 + 0.30×1.0 + 0.40×1.0 = 1.000

ToDo T2: 回复 Alice 确认参会  (email_draft)
├── L1 执行结果: 1.0 (成功生成邮件草稿)
├── L2 过程质量: 0.5 × 0.75 + 0.5 × 1.0 = 0.875
│   ├── InfoUtil: 3/4 = 0.75 (未利用日历结果)
│   └── ToolQuality: 1.0 (无额外工具调用)
└── L3 交付物质量: 0.25 × 1.0 + 0.75 × 0.80 = 0.85
    ├── Format: 1.0 (全部检查通过)
    └── Rubric: 8/10 = 0.80
→ S_T2 = 0.30×1.0 + 0.30×0.875 + 0.40×0.85 = 0.903

ToDo T3: 预订 10 人会议室  (tool_action)
├── L1 执行结果: 1.0 (book_room 返回成功)
├── L2 过程质量: 0.5 × 0.80 + 0.5 × 0.90 = 0.85
└── L3 交付物质量: = ExecSuccess = 1.0
→ S_T3 = 0.30×1.0 + 0.30×0.85 + 0.40×1.0 = 0.955

ToDo T4: 准备 Q1 PPT 初稿  (document)
├── L1 执行结果: 1.0 (生成了大纲文档)
├── L2 过程质量: InfoUtil = 0.60 (无工具调用)
└── L3 交付物质量: 0.25 × 0.80 + 0.75 × 0.70 = 0.725
    ├── Format: 0.80
    └── Rubric: 7/10 = 0.70
→ S_T4 = 0.30×1.0 + 0.30×0.60 + 0.40×0.725 = 0.770

S_B = (1.000 + 0.903 + 0.955 + 0.770) / 4 = 0.907

5. 评估模块 C — 稳定性

5.1 评估方法

对同一输入重复运行 Agent k 次（推荐 k=5），观测输出变异。

M9: 一致性 pass^k参考 τ-Bench [9] pass^k

$$pass^k = \frac{1}{N} \sum_{i=1}^{N} \mathbb{1}[\text{任务 i 在全部 k 次运行中均成功}]$$

与 pass@k（至少一次成功）相对，pass^k 衡量的是"每次都成功"的严格一致性，更适合生产环境。

M10: 输出稳定性（Output Stability）参考 Survey [5] Consistency 章节

$$Stability = \frac{2}{k(k-1)} \sum_{i < j} sim(output_i, output_j)$$

M11: 鲁棒性（Robustness）参考 HELM [5, Ref 51]; Survey [5] Robustness 章节

扰动类型方法
措辞改写用 LLM paraphrase 邮件正文
信息噪声添加无关段落或签名信息
格式变化HTML 邮件 ↔ 纯文本
语言混用中英文混合邮件
$$Robustness = \frac{S_{perturbed}}{S_{original}}$$

5.2 模块 C 综合分数

$$S_C = 0.40 \times pass^k + 0.30 \times Stability + 0.30 \times Robustness$$

6. 整体评估流程

6.1 流程总览

Phase 0: 数据集准备与验证
  ├─ Step 0a: 定义 task_type_rules（一次性）
  ├─ Step 0b: 标注 gold_todos, gold_tool_calls（人工）
  ├─ Step 0c: 生成并审核 gold_key_info, rubrics（LLM + 人工）
  └─ Step 0d: 验证 Coverage / IAA / Discriminability ≥ 0.75

Phase 1: 单次运行评估（每个 test case 执行 1 次）
  ├─ Step 1: Agent 接收邮件输入
  ├─ Step 2: 记录 Agent trace（工具调用、中间推理）
  ├─ Step 3: 收集 Agent 输出（ToDo list + 执行过程 + Deliverables）
  ├─ Step 4: 模块 A — ToDo 识别评估 (M1-M3)
  ├─ Step 5: 模块 B — 逐 ToDo 端到端执行质量 (M4-M8)
  └─ Step 6: 汇总单次运行结果

Phase 2: 稳定性评估（每个 test case 执行 k 次）
  ├─ Step 7: 重复运行 k=5 次
  ├─ Step 8: 计算 pass^k, Stability (M9-M10)
  └─ Step 9: 扰动测试计算 Robustness (M11)

Phase 3: 综合评分与报告
  ├─ Step 10: 按难度分层统计
  ├─ Step 11: 计算总分
  └─ Step 12: 生成评估报告（含 error analysis）

6.2 总分公式

$$S_{total} = 0.30 \times S_A + 0.50 \times S_B + 0.20 \times S_C$$

6.3 按难度分层报告

层级说明权重
Tier 1: Explicit关键词和意图直接匹配0.20
Tier 2: Semantic需要语义推理0.40
Tier 3: Adversarial含混淆/对抗性元素0.40
$$S_{weighted} = \sum_{t} w_t \times S_{total}^{(t)}$$
